GROCERY LIST HELPER - BACKEND API NOTES

Base URL:
http://localhost:8080

General notes:
- IDs used in URLs must already exist in the database.
- Date format for pantry expiryDate must be:
  YYYY-MM-DD
  Example: 2026-08-01
- Change root password--default is set to "root"
- Validation works on backend request bodies.
- If invalid input is sent, backend returns field-specific validation errors as JSON.

==================================================
VALIDATION CURRENTLY USED
==================================================

The backend now uses validation annotations on DTOs.

Main validation rules used:
- @NotBlank
  Used for required text fields like:
  - name
  - email
  - password
  - list title
  - item name
  - ingredient name

- @NotNull
  Used for required non-text fields like:
  - userId
  - listId
  - ingredientId
  - quantity
  - bought

- @Positive
  Used for numeric values that must be greater than 0:
  - quantity

- @Email
  Used for email format validation

- @Size(min = 4)
  Used on register password so password must be at least 4 characters

- @Valid
  Used for nested request bodies like:
  - full shopping list creation route
  - item lists inside full list create
  - ingredient link lists inside full list create

If validation fails, backend returns JSON field errors like:
{
  "email": "Email format is invalid",
  "password": "Password must be at least 4 characters"
}

Business logic checks are also still used in services, for example:
- Email already exists
- User not found
- Shopping list not found
- Ingredient not found
- Ingredient already exists

==================================================
1. AUTH
==================================================

A. Register User
Method:
POST

Endpoint:
http://localhost:8080/api/auth/register

Purpose:
Creates a new user account.

Request body example:
{
  "name": "Christian",
  "email": "christian@email.com",
  "password": "1234"
}

Expected success response:
User registered successfully

Possible business logic responses:
- Email already exists

Validation used:
- name -> required
- email -> required + valid email format
- password -> required + minimum 4 characters

Example invalid request:
{
  "name": "",
  "email": "not-an-email",
  "password": ""
}

Example validation response:
{
  "name": "Name is required",
  "email": "Email format is invalid",
  "password": "Password must be at least 4 characters"
}

--------------------------------------------------

B. Login User
Method:
POST

Endpoint:
http://localhost:8080/api/auth/login

Checks whether the user exists and whether the password matches.

Request body example:

Example validation response:
{
  "email": "Email is required",
  "password": "Password is required"
}

B. Create Full Shopping List (list + items + ingredients)
Method:
POST

Endpoint:
http://localhost:8080/api/lists/full

Purpose:
Creates a shopping list and also creates its shopping items and linked ingredients in one request.

Request body example:
{
  "title": "Frontend Test List",
  "userId": 3,
  "items": [
    {
      "itemName": "Milk",
      "quantity": 2,
      "category": "Dairy"
    },
    {
      "itemName": "Apples",
      "quantity": 5,
      "category": "Fruit"
    }
  ],
  "ingredients": [
    {
      "ingredientId": 2,
      "quantity": 12
    }
  ]
}

Expected success response:
{
  "message": "Shopping list with items and ingredients created successfully",
  "listId": 9
}

Possible business logic responses:
{
  "message": "User not found",
  "listId": null
}

{
  "message": "List title cannot be empty",
  "listId": null
}

Validation used:
- title -> required
- userId -> required
- each item itemName -> required
- each item quantity -> required + positive
- each ingredient ingredientId -> required
- each ingredient quantity -> required + positive

Example invalid request:
{
  "title": "",
  "userId": null,
  "items": [
    {
      "itemName": "",
      "quantity": 0,
      "category": "Dairy"
    }
  ],
  "ingredients": [
    {
      "ingredientId": null,
      "quantity": 0
    }
  ]
}

Possible validation response example:
{
  "title": "List title is required",
  "userId": "User ID is required"
}

Notes:
- This route was added so frontend can create a full grocery list in one action instead of multiple separate requests.

{
